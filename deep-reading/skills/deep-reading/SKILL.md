# 深度阅读

## 核心理念

深度阅读将一本书转化为两个互补的结构：

1. **把书读薄**（静态）：按原书目录层级建立文件夹结构，在每个小节创建阅读笔记，忠实跟随原书框架
2. **实体概念沉淀**（动态）：将阅读中涌现的关键概念提取为"概念卡片"，卡片末尾用分割线隔开延伸方向，通过 `[[wikilink]]` 链接到其他概念或笔记，逐步生长为概念网络

两部分共同在 Obsidian 中形成可双向链接、可图谱可视化的知识体系。

## 两种结构的关系

- **目录笔记**是土壤：忠实地跟随原书结构，把每个小节的要点、摘录、批注沉淀下来。文件放在按原书层级创建的章节文件夹中。
- **实体概念卡片**是生长出来的树：在阅读过程中不断冒出，**文件放在 `实体概念/` 文件夹下（扁平，不建子文件夹）**，卡片之间通过 wikilink 互链，逐渐织成跨章节甚至跨书的概念网络。
- 一张概念卡片最初可能只有一个定义，随着阅读推进，它的延伸方向会不断丰富

## 重要：不要混淆两种文件

- 章节文件夹下的 `.md` 文件是小节阅读笔记，使用"阅读笔记模板"
- `实体概念/` 下的 `.md` 文件是概念卡片，使用"概念卡片模板"
- 它们是完全不同的两类文件，放在不同的位置，使用不同的格式
- 即使用户说"这个概念来自XX小节"，概念卡片也应放在 `实体概念/` 下，而不是放在章节文件夹中

## 整体工作区视角

深度阅读以单个**主题工作区**为顶层容器。一个工作区可以容纳多本互相关联的书。每本书独立管理自己的目录笔记和实体概念，跨书概念通过 wikilink 互链。

工作区的 CLAUDE.md 负责记录：
- 每本书的路径、作者、类型、一句话说明
- 阅读进度和实体概念数量
- 跨书主题阅读的入口

## 第一部分：把书读薄 —— 目录与笔记

### 1.1 初始化一本书

当用户说"初始化《XXX》"时，执行以下步骤：

**第〇步：确定书名和文件夹名**

"书名"有两个可能来源：epub/md 文件内部提取的正式书名，以及文件名的简称。初始化时优先使用**文件内部提取的正式书名**（作为 `book` frontmatter 字段和 `_目录.md` 标题），但文件夹名可使用**文件名简化版**（去掉作者、出版社等后缀）。SYNC 标记的 key 必须与文件夹名一致。如果两个名字不同（如 epub 内的"零零年代"与文件名中的"〇〇年代"），在 CLAUDE.md 的说明字段中注明。

**第一步：获取目录信息**

直接读取原书文件提取章节目录。根据文件类型选择方式：
- **epub 文件**：用 `pandoc <文件路径> -t plain --wrap=none 2>&1 | head -200` 提取开头部分，从章节标题（如"第一章""PART 1"等）识别目录结构。若全文章节较多，用 `pandoc <文件路径> -t plain --wrap=none 2>&1 | grep -n "^第.*章\|^PART\|^CHAPTER"` 提取所有章节标题行号。
- **Markdown 文件**：直接 Read 文件，从标题层级（`#`、`##`、`###`）解析目录结构
- **PDF 文件**：先转换为 Markdown 再解析

⚠️ **章节匹配防误判**：`grep -n "^第.*章"` 可能匹配到内文中引用章节号的句子（如"第一章已经指出..."）。匹配结果需要二次审核：
- 真正的章节标题通常独占一行、行内容以"第X章"开头后紧跟空格或标点，行内无其他内容片段
- 内文引用通常以完整句子开头（如"第一章已经指出了..."），不会紧跟在"第X章"之后换行。真正的章节标题行后面通常没有"已经""笔者""我们"等后续词的句子
- 如果某行匹配到 `^第` 但该行的行号与前一个匹配结果不在合理间隔内（例如相距仅几十行且之前已有同级章节），应检查是否为内文引用

提取的结果包括：每本书的完整层级结构（编→章→节）、每个小节的行号范围（用于后续 pandoc 定位）。行号信息记录在 `_目录.md` 中。

**第二步：创建文件夹结构**

在书籍文件夹下，完全按照原书的层级创建目录结构。每一个层级都保留原书的编号和标题。不限制层级深度——原书有几层就建几层。每一层级的文件夹/文件名应保留原书的编号前缀（如"第一章""PART1""CHAPTER01"等）。

**第三步：创建小节 .md 文件**

最小笔记单元是小节（原书最末级的小标题）。每个小节创建一个 .md 文件，使用以下**阅读笔记模板**。

如果原书只有章层级、没有细分小节（如某些评论集/专著），则每章作为一个笔记文件，放在对应章节文件夹下。文件名用章的简化标题，section frontmatter 字段填章标题。

```markdown
---
tags: [阅读笔记]
book: "<书名>"
chapter: "<章标题>"
section: "<节标题>"
status: unread
date: <当前日期>
---

# <节标题>

## 摘要

（待阅读后填写）

## 关键要点



## 原文摘录



## 个人批注



## 本节概念

```

注意：初始化时只创建上述骨架模板，不要自行填充内容。内容在后续用户实际阅读时再补充。

导航字段的层次结构因书而异。例如：
- 《文学批评入门》：编 → 章 → 节（section 填节标题，chapter 填章标题）
- 《写作技法荟萃》：技法大类 → 技法条目（section 填技法名，chapter 填技法大类）
- 《麦基三部曲》：书 → PART → CHAPTER → 小节

**第四步：创建实体概念文件夹**

在书籍文件夹根目录下创建 `实体概念/` 空文件夹（仅此一个），用于后续动态沉淀概念卡片。

**禁止**在任何章节文件夹（如"立意技法/"、"第一章-.../"等）下再创建 `实体概念/` 子文件夹。实体概念文件夹只在书根目录下存在一个。

**第五步：更新 CLAUDE.md**

在工作区 CLAUDE.md 的书籍清单中添加该书的条目，使用 SYNC 标记包裹同步字段：

```markdown
### <书名>
<!-- SYNC:BOOK:<书名> -->
- **路径**：`<书名>/`
- **作者**：<作者>
- **类型**：<教材/专著/辞典等>
- **说明**：<一句话描述>
- **阅读进度**：0个小节中 0个已完成，0个进行中，0个未读
- **阅读状态**：未开始
- **实体概念数**：0
<!-- SYNC:END -->
```

`<!-- SYNC:BOOK:书名 -->` 和 `<!-- SYNC:END -->` 是不可见标记，让自动同步脚本能精确匹配和更新同步块，而不依赖脆弱的自然语言正则。"阅读进度"是统计数字（脚本自动更新），"阅读状态"是语义描述（AI 推断更新）。

### 1.2 创建目录导航文件

在每本书的根目录创建一个 `_目录.md` 文件。这是全书唯一的索引导航文件，作用是让 AI 知道：**文件在哪里、怎么读、每个小节对应哪些行号、笔记链接是什么**。

#### 必须包含的元信息

文件开头必须包含：

```markdown
# 《书名》目录

> **作者**：<作者名>
> **文件路径**：`<原书绝对路径>`
> **读取方式**：<直接 Read Markdown / 通过 pandoc 转换 epub 后按行号读取>
> **命令行**：<可复制执行的命令模板，用 `<起始行>` `<结束行>` 作为占位符>
```

- **文件路径**：原书的绝对路径，让 AI 无需搜索即可定位
- **读取方式**：说明用哪种工具读取。Markdown 文件填"直接 Read 按行号跳转"；epub 文件填"通过 pandoc 转换为纯文本后按行号读取"
- **命令行**：提供可复制执行的命令模板。epub 格式：`` pandoc "<文件路径>" -t plain --wrap=none 2>&1 | sed -n '<起始行>,<结束行>p' ``；Markdown 格式：`` Read file_path="<文件路径>" offset=<起始行> limit=<行数> ``

#### 目录结构

目录使用 Markdown 标题层级（`##`、`###`、`####`）反映原书结构，用表格列出每个小节的行号和 wikilink。

**表格式目录**：

```markdown
## 一、立意技法（行 240–673）

| 行号 | 技法 |
|------|------|
| 242 | **[[立意技法/画龙点睛\|画龙点睛]]** |
| 262 | **[[立意技法/渐远渐淡\|渐远渐淡]]** |
| 280 | **[[立意技法/卒章显志\|卒章显志]]** |
| ... | ... |
```

小节的结束行 = 下一小节的起始行 − 1。最后一个小节的结束行 = 父级章节的结束行。

**嵌套列表式目录（备选，当原书没有统一表格结构时使用）**：

```markdown
## 第一编 什么是学院派批评

| 行号 | 章节 |
|------|------|
| 299 | 第一编导语 |
| 304 | **第一章 学院里的批评家** |
| 308 | [[第一编-什么是学院派批评/第一章-学院里的批评家/为什么需要批评\|为什么需要批评]] |
| 328 | [[第一编-什么是学院派批评/第一章-学院里的批评家/当批评遭遇理论\|当批评遭遇理论]] |
| ... | ... |
```

#### 行号规则

- 表格中每行的行号是**该小节的起始行**
- 该小节的结束行 = 下一行的行号 − 1
- 最后一个小节的结束行 = 父级章节标注的行号范围上限（如"行 240–673"中的 673）
- AI 读取原文时，读取范围是 `起始行` 到 `结束行`

### 1.3 合订本处理规则

当源文件是套装/合订本（如《罗伯特·麦基虚构艺术三部曲》）时，**必须先询问用户**：
- "这本书是套装/合订本，你希望合在一起当一本书处理，还是拆开当多本独立书？"

根据用户选择创建对应的文件夹结构。拆开时，每本书各自拥有独立的目录层级和实体概念文件夹。

### 1.4 做阅读笔记

当用户说"阅读 XXX 小节"或"做 XXX 的笔记"时：

1. **定位原文**：读取该书的 `_目录.md`，从对应小节的行号获取原文范围，然后用 pandoc + sed（epub）或直接 Read（md）读取原文内容
2. **写入笔记**：在小节 .md 文件中填充内容，使用以下模板：

```markdown
---
tags: [阅读笔记]
book: "<书名>"
chapter: "<章标题>"
section: "<节标题>"
status: reading
date: <当前日期>
---

# <节标题>

## 摘要
（本节核心内容概括，2-3句话）

## 关键要点
- 要点1
- 要点2
- ...

## 原文摘录
> 值得记住的原文段落

## 个人批注
（用户自行填写，或留下空白供后续补充）

## 本节概念
- [[../实体概念/<概念名>|<概念名>]]
```

**重要原则**：
- "摘要"和"关键要点"应忠实于原文，避免过度解读
- "个人批注"留给用户自行填写
- "本节概念"自动列出本节涉及的实体概念 wikilink（如果已创建对应概念卡片）
- 将 frontmatter 的 `status` 更新为 `reading`（进行中）或 `completed`（已完成）

### 1.5 更新阅读进度

每次做完阅读笔记后，更新两个地方：
1. 该小节文件的 `status` frontmatter 字段
2. CLAUDE.md 中对应书的 SYNC 块（阅读进度数字和阅读状态）

### 1.6 自动同步到 CLAUDE.md

每次完成以下操作后，自动更新 CLAUDE.md 中对应书籍的 `<!-- SYNC:BOOK:... -->` 块：

**更新阅读进度统计**：
- 扫描该书中所有小节 .md 文件的 `status` frontmatter
- 统计 unread / reading / completed 数量
- 更新 SYNC 块中的"阅读进度"为：`X个小节中 Y个已完成，Z个进行中，W个未读`

**更新实体概念数**：
- 统计 `实体概念/` 下的 .md 文件数量
- 更新 SYNC 块中的"实体概念数"

**更新阅读状态**（语义描述，AI 推断）：
- 根据阅读进度统计和最后一次操作推断并更新"阅读状态"
- 全部 unread → "未开始"
- 有 completed 且无明显模式 → "正在阅读"
- 某章节全部 completed → "已完成XX章"
- 全书全部 completed → "已读完"

**更新双向链接**：
- 如果创建或更新了概念卡片，检查概念卡片的 `source` frontmatter 指向的笔记文件是否存在
- 如果做完了阅读笔记（status 从 unread 变为 reading/completed），检查"本节概念"区的 wikilink 目标是否存在

## 同步两层架构

深度阅读的同步分为两个独立层，互补而不依赖：

1. **脚本层**（Stop hook → auto-sync.py）：对话结束时由系统自动触发 Python 脚本，扫描文件系统做数字字段更新。轻量、可靠、不消耗 token。脚本失败不影响正常使用。
2. **AI 层**（Skill 1.6 节）：在每次阅读笔记/概念卡片操作后，由 AI 做语义判断和交叉验证。需要理解力。

两层分工明确：脚本层只做确定性操作（文件计数、数字替换），AI 层做需要理解力的操作（语义描述推断、双向链接交叉验证）。

## 第二部分：实体概念沉淀 —— 概念卡片与延伸

### 2.1 概念卡片是什么

概念卡片是一个 .md 文件，**必须放在书籍的 `实体概念/` 文件夹下**，不放任何子文件夹，与其他所有概念卡片平级。

概念卡片包含两个区域，用分割线 `---` 隔开：

- **分割线以上**：概念本体 —— 核心定义、详细展开、来源引文
- **分割线以下**：延伸方向 —— 这个概念与其他概念/知识点的关联网络

### 2.2 创建概念卡片

当用户说"创建概念卡片 XXX"或"提取概念 XXX"时，在 `<书籍>/实体概念/` 下创建 `<概念名>.md` 文件。

⚠️ **关键**：概念卡片必须放在 `实体概念/` 下，不是放在章节文件夹中。即使概念来源于某个特定小节，文件也放在 `实体概念/` 下，通过 `source` frontmatter 指向来源。

使用以下**概念卡片模板**：

```markdown
---
tags: [实体概念, <领域标签>]
book: "<书名>"
source: ["[[<来源章节路径>]]"]
aliases: [<英文名/别名>]
related: ["[[<关联概念>]]"]
status: draft
date: <当前日期>
---

# <概念名>

## 核心定义
```

**领域标签参考**：叙事学、剧作理论、写作技法、批评理论、文体学、人物理论、对白理论、修辞学

**常见错误**：
- ❌ 把概念卡片放在章节文件夹下（如"立意技法/画龙点睛.md"）→ 正确位置是"实体概念/画龙点睛.md"
- ❌ 使用阅读笔记模板来写概念卡片 → 两个模板不同，不要混淆
- ❌ 在延伸方向中只写概念名不写说明 → 每个关联概念都需要1-2句简要说明

### 2.3 延伸方向的写法

延伸方向中的每个条目遵循：**wikilink + 简要说明**。简要说明控制在1-2句话，说清楚这个关联是什么。

对于简单关联（一句话能说清），直接写在概念卡片的延伸方向区。

对于复杂关联（需要多维度论述、实例说明、或涉及多本书的交叉），创建独立的**关系文档**。关系文档是一个普通的 .md 文件，可放在 `实体概念/` 下，用描述性文件名。

### 2.4 迭代生长

概念卡片不是一次性完成的。随着阅读推进：
- 新的概念卡片被创建
- 已有概念卡片的"延伸方向"区被更新（添加新发现的关联）
- 关系文档在关联变得复杂时被创建
- 概念的 `related` frontmatter 字段同步更新

当用户说"更新延伸方向 XXX"时，扫描该书中所有已读的笔记和概念卡片，找出提及或关联该概念的地方，补充到延伸方向区。

### 2.5 概念卡片与笔记的双向链接

- 在章节笔记的"本节概念"中，用 wikilink 指向概念卡片
- 在概念卡片的 "source" frontmatter 中，用 wikilink 指向来源章节
- 在 Obsidian 中打开任意概念卡片，Backlinks 面板会自动展示所有提及该概念的笔记

## 第三部分：跨书概念与主题阅读

### 3.1 跨书概念卡片

当用户想要跨书整合某个概念时，在工作区根目录下创建跨书概念卡片：

```
<工作区>/
├── 《书A》/
├── 《书B》/
├── 跨书概念/
│   └── <概念名>.md    ← 跨书统一卡片
```

跨书概念卡片的格式与普通概念卡片相同，但在"来源与引文"中综合了多本书的视角，在"延伸方向"中会引用各书的概念卡片。

### 3.2 工作区级阅读感想

用户可能在阅读多本书后产生主题级的感想。此时更新 CLAUDE.md 的"跨书主题阅读"部分，记录感想的摘要和指向具体笔记的链接。

## 与 Obsidian 集成

深度阅读 skill 产出的所有文件都是标准 Markdown + wikilink + YAML frontmatter。用 Obsidian 打开工作区文件夹作为 vault 即可获得：

- **Graph View**：自动展示概念网络图谱，每个文件是一个节点，`[[wikilink]]` 是边
- **Backlinks 面板**：打开任意文件，看到所有引用它的文件
- **标签导航**：通过 `tags` frontmatter 筛选阅读笔记 vs 实体概念
- **Dataview 插件（可选）**：用 dataview 查询管理阅读进度

无需额外工具或 Obsidian 插件即可工作。Obsidian CLI 是可选的增强工具（需要 Catalyst 许可证），但核心功能不依赖它。

## 注意事项

- **目录层级不截断**：完全按照原书的层级结构创建文件夹，不要自行压缩或合并层级
- **概念文件夹扁平**：`实体概念/` 下的所有概念卡片直接平放，不建子文件夹
- **wikilink 使用相对路径**：从当前文件位置出发，确保 Obsidian 能正确解析
- **source frontmatter 始终指向笔记**：每张概念卡片的 `source` 字段应指向具体的阅读笔记文件，建立来源可追溯
